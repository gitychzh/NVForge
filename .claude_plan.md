# R825: ms_gw HTTP 400 风暴 → 86-attempt 5分钟卡死 系统性修复

> 用户报 "cx4102 fallback 错误: all keys exhausted for model=glm5_2_ms ... http_400 ×N"。
> 远程 HM2 实测 + 源码深挖定位。改 HM2 的 ms_gw(外科 patch,不整覆盖)。
> 铁律: 改前有数据 (日志+实测), 改后有验证 (端到端)。

## 现象 (2026-07-08, 远程 HM2)

用户用 Claude Code (实际走 cx4102:4102 或 cc4101:4101), 报错:
```
⚠️ [cx4102 fallback 错误] ms_gw: all keys exhausted for model=glm5_2_ms.
Attempted: v9k1=stream_rl_backoff:http_429_rate_limit; v9k2=...;
v9k3=stream_cycle:http_400; v9k4=stream_cycle:http_400; ...(大量 http_400)
v0k5=stream_rl_backoff: (reason为空)
```

实测 (docker logs):
- b28c524a 请求穿越 10 variant × 7 key = **86 attempts**, 16:24:48→16:29:30 = **4分42秒卡死**
- 同 key+variant 时而 429 (backoff) 时而 400 (cycle) 时而 200 → 400 是**瞬时后端故障**, 非单 key 永久问题
- cx4102 FALLBACK_TIMEOUT_S=300s 给了完整跑完 70 槽的时间窗
- nv_gw 同期持续 502 (NVCF func 3b9748d8 glm5_2_nv + 74f02205 dsv4p_nv 双双 504/DEGRADING) → 全量流量压到 ms_gw

## 根因链 (5 层, 已逐层验证)

### 根因 1 (上游, 非代码bug): NVCF 双 function 故障
nv_gw 的 glm5_2_nv (func 3b9748d8) + dsv4p_nv (func 74f02205) 持续 504/timeout/DEGRADING,
peer-fb skip 后返回本地 502, 所有 agent (cx4102/cc4101) 全量 fallback 到 ms_gw。
→ 流量翻倍压到 ModelScope。

### 根因 2 (真根因, 诊断patch抓到body): ModelScope 对 `developer` role 返回 400
诊断 patch (Fix-D) 抓到 400 真实 body:
```json
{"error":{"code":"invalid_parameter_error",
"message":"developer is not one of ['system','assistant','user','tool','function']",
"type":"invalid_request_error"}}
```
**根因**: Codex 客户端发 OpenAI Responses API 的 `role:"developer"` 消息(developer 是 system 的别名),
cx4102 codex.py 行55 `role = item.get("role","user")` 原样透传 → ms_gw sanitize 只删 thinking 参数不动 role
→ ModelScope/GLM 不认 developer role → 确定性 400。换 key/variant 都 400 (请求体问题,非后端故障)。
健康期直连 7 key 全 200 (实测 15 并发全 200, 带 30 tools/41 msgs/thinking 残留参数都 200)。
注: 另有一类瞬时 400 (后端故障期, 同 key 时而400时而200), 但当前用户报错的主因是 developer role 确定性 400。

### 根因 3 (核心 BUG, ms_gw upstream.py): 400 被当作单 key 故障 → mark_key_cooling
`_check_ms_stream_first_chunk` (upstream.py:196-270) 对非200分支:
```python
if resp_status != 200:
    if resp_status == 429 or any(c in body_hint for c in rl_codes):
        is_rate_limit = True  # → backoff, key kept warm ✓
    result.cycle = True
    result.reason = "http_%d" % resp_status  # 400 走这里
```
400 不匹配 rl_codes (除非 body 含 insufficient_quota 字样), 走 `cycle` →
`_try_ms_keys` 行 468 `mark_key_cooling(model, v, k)` 冷却该 key 60s (指数→封顶30s)。

**错误**: 400 是后端级瞬时故障 (同 key 下一秒就 200), 却按"该 key 坏了"冷却 30-60s。
→ 一个请求 cycle 穿越所有 70 槽, 每槽 mark_cooling, 最终 all_exhausted 503。
→ 冷却的 key 在**后续请求**也被 skip, 可用槽位减少, 扩散到更多请求触顶。

### 根因 4 (放大器, ms_gw config): MIN_OUTBOUND_INTERVAL_S=2.0 + 429 backoff 3s
每次 outbound 强制间隔 2s (rpm=1 保护), 429 再 backoff 3s。
86 attempts × ~3.5s = ~300s。加上 400 也会等满 2s 节流。→ 5分钟卡死主因之一。

### 根因 5 (无提前中止, ms_gw upstream.py): http_400 不计入 total_empty cap
`EMPTY_200_TOTAL_LIMIT=12` (行 360) 只对 `chk.empty_200` 计数 (choices_null/content_empty 等)。
http_400 走 cycle 但 `chk.empty_200=False`, **不计入 total_empty**, 无法触发提前中止。
→ 400 风暴必须跑完 70 槽或踩满 fastbreak(每 variant 连续5次empty, 但400不算empty)才停。
日志证实: b28c524a 跑了 86 attempts 才因 variant 全冷却而 all_exhausted。

## 修复方案 (分层, 最小侵入)

### Fix-A (核心): ms_gw 把 HTTP 400/5xx 当瞬时故障 backoff, 不 mark_key_cooling
**文件**: /opt/cc-infra/proxy/ms-gw/gateway/upstream.py (`_check_ms_stream_first_chunk` + `_try_ms_keys`)

把非200响应统一归类:
- 429 / body 含 rate_limit/insufficient_quota/limit_burst → `is_rate_limit=True`, backoff 3s, key kept warm (现状, 保留)
- **400 / 5xx / 其他非200 → 新增 `is_transient=True`**, 走 backoff (较短, 1-2s), **不 mark_key_cooling**, continue
  - 理由: 400 是后端瞬时故障 (同 key 下一秒恢复), 冷却 key 反而害后续请求
  - 与 429 backoff 区分: 429 是配额限流(长 backoff 3s), 400 是后端抖动(短 backoff)
- 只有 **empty_200 (choices_null/content_empty/error_body:insufficient_quota)** 这类"该 key 确实配额耗尽/坏"才 mark_key_cooling (现状保留)
  - insufficient_quota 在 body 里时也算 key 级耗尽 → cooling (修正: 当前 insufficient_quota 在 rl_codes 里走 backoff 不 cooling, 但它其实是配额耗尽应 cooling — 见 Fix-A 细节)

**实现**: 给 MSResponseCheck 加 `is_transient` 字段; `_check_ms_stream_first_chunk` 非200且非rate_limit时设 `is_transient=True,cycle=True`; `_try_ms_keys` 流式+非流式 cycle 分支在 `is_transient` 时走 backoff 路径(不 cooling, 短 sleep), 仅 `empty_200/error_body` 才 mark_key_cooling。

### Fix-B (提前中止): http_400/5xx 计入 total cap
**文件**: 同 upstream.py
把 `total_empty` 计数扩展为 `total_cycle` (或新增 `total_transient`), http_400/5xx/transient 也计数,
达到 `EMPTY_200_TOTAL_LIMIT` (建议改名 `CYCLE_TOTAL_LIMIT` 或复用) 即提前中止 → all_exhausted。
这样即使 backoff 不 cooling, 也不会无限重试 70 槽, 而是快速 fail 让 cx4102 收到 503 (而非 5 分钟后)。
**配合 Fix-A**: transient backoff 1-2s + cap 12 次 ≈ 12-24s 触顶, 而非 300s。

### Fix-C (止血, cx4102): FALLBACK_TIMEOUT_S 300→60
**文件**: /opt/cc-infra/proxy/cx4102 (env 或 config.py)
300s 给了 ms_gw 完整跑完 70 槽的时间窗。降 60s: ms_gw 即使没 Fix-A/B, 也最多卡 60s 而非 5 分钟。
(Claude Code/Codex 客户端通常 60-90s 超时, 300s 远超用户耐心。)
注: cc4101 已是 TOTAL_BUDGET=80s, 问题轻; cx4102 是重灾区。

### Fix-D (已完成): R825-diag patch 抓到 400 真实 body → 定位 developer role 真根因
诊断 patch 抓到 400 body = `invalid_parameter_error: developer is not one of [...]`。
→ 由此定位真根因 (根因 2), 实施 Fix-E (role 归一化)。
诊断 body 记录功能 (stream_non200_body) 保留在 upstream.py (低频, 运维诊断价值), 不还原。
.bak.diag400 / .bak.preFix 保留作回滚点。

### Fix-E (真根因根治, 已实施): developer role → system 归一化
- cx4102 codex.py 行55: `role=="developer" → "system"` (源处转换)
- ms_gw upstream.py `_sanitize_request_body`: 兜底归一化, developer + 任何非标准 role → system
- 双层防御: cx4101 (Anthropic, 无 developer) 不受影响; 任何走 ms_gw 的 developer role 都被兜底
- 验证: 直接打 ms_gw 带 developer role → 一次 200 + 日志 `MS-SANITIZE role developer→system` + `MS-OK-STREAM`;
  端到端 cx4102 Responses API 带 developer role → 200 流式 (primary 502 fallback ms_gw 成功)

### Fix-E (可选, 后续): nv_gw NVCF 双 function 故障的独立处理
根因 1 是上游 NVCF 故障, 非本次代码 bug。但 cx4102/cc4101 在 primary(nv_gw) 持续 502 时,
circuit breaker 已能跳过 primary 直 fallback (cx4102 CIRCUIT_OPEN_S=60, cc4101 SKIP_S=120)。
本次不改 nv_gw。若 NVCF 长期不恢复, 单独排期。

## 验证

1. **Fix-D 先抓 400 body** (诊断 patch 已跑): 等下次 ModelScope 故障期, `docker exec ms_gw grep stream_non200_body /app/logs/ms_error_detail.2026-07-08.jsonl` 确认 400 真实内容 → 校准 Fix-A 分类
2. **Fix-A/B 改后**: 构造 400 风暴 (或等故障期) → 观察 ms_gw 日志: 400 应走 `MS-RL-BACKOFF` 类短 backoff (新增 tag 如 `MS-TRANSIENT-BACKOFF`), 不再 `MS-STREAM-CYCLE`+mark_cooling; 单请求 attempts 应 ≤12 (cap) 而非 86
3. **端到端**: cx4102 fallback 到 ms_gw, 即便 ms_gw 触顶也应 60s 内返回 503 (Fix-C) 而非 300s; 用户不再卡 5 分钟
4. **健康期回归**: 15 并发全 200 (已实测), 改动不破坏健康路径 (transient 分支只在非200触发)
5. **冷却快照**: `/health` 的 keys_cooling 在故障期应大幅减少 (400 不再 cooling), 恢复后全清
