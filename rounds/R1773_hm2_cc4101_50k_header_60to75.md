## 根因分析
- cc4101 `upstream.py` line 216-217: 50K-200K input 档 `_hdr_to = 60` (硬编码, 非 env)
- nv_gw `handlers.py` line 893: 50K-200K 档 `NVU_STREAM_FB_50K_S=60` (first_byte deadline)
- **两个 60s 完全撞一起**: cc4101 connect+ttfb 60s 抢先到 → 切 ms_gw, nv_gw 这条 first_byte 60s deadline 形同虚设 (cc4101 已切走上游连接). nv_gw 这条自己继续等上游 SSE 74-126s 才 report stream_first_byte_timeout — 但 cc4101 那头 fallback 已 OK, 数据空洞形成.
- R1638 当时已修 200-350K 档 90→120 对齐 nv_gw chain budget, 但 **50K-200K 档遗漏**, 仍 60s 与 nv_gw first_byte 60s 倒挂.
- STATE 下一轮建议原文: "若连续窗口 >2%, 再回头评估 50K 档首字节阈值 (小步: 60s→70s)". 本轮即按此执行 — 但发现真正要改的是 cc4101 这条 60s (nv_gw 的 60s 被 cc4101 抢先用不到), 故改 cc4101 60→75.

## 拟改
- **文件**: `/opt/cc-infra/proxy/cc4101/gateway/upstream.py` line 216-217
- **改动**: `elif _hdr_ic > 50000: _hdr_to = 60` → `_hdr_to = 75`
- **理由**: 75s = nv_gw 50K-200K 档 first_byte 60s deadline + 15s 余量. 让 nv_gw 先用满自己的 60s first_byte 检测 (上游真首字节慢则 nv_gw 主动 break 发 err_chunk 干净 Scenario A, cc4101 75s 未到正常会计数 polarity), 再让 cc4101 兜底. 不让 cc4101 抢先 cut.
- **不改 nv_gw (40006)**: nv_gw first_byte 60s 保留 (它是真实"上游首字节健康"探测器)
- **不碰 ms_gw (40007)**: 仍作 restart 窗口热备, 改的是 cc4101 调度逻辑
- **备份**: `upstream.py.bak.R1772`(改前已存, 用 R1772 备份名因当时以为用 R1772 轮号, 后 peer 抢占改用 R1773 — 备份文件名不影响内容正确性)

## 预期
- 50K-200K 档 PRIMARY-FAIL header/ttfb timeout 从 60s 推到 75s 触发 → fallback 率降 (上游若 60-75s 出首字节则 cc4101 收到不再切 ms)
- nv_gw 真 stream_first_byte_timeout (上游 75s 仍无首字节) 仍会被 nv_gw 自己 detect break, 干净失败不计 breaker 误 OPEN
